# Naming Principles

The foundational rules for what makes a name work. These are not preferences — they're patterns backed by research, brand history, and how human memory actually functions.

---

## 1. Metaphor Is King

The most memorable names map a **concrete metaphor** to an abstract function. Not a description — a metaphor.

A metaphor gives the listener something to see. The brain stores images far more reliably than it stores labels. When someone hears "Sentry," they see a guard at a gate. When someone hears "SiteWatch," they see... nothing. A description and a metaphor can both communicate what a product does, but only the metaphor is memorable.

**The test:** Can you draw a picture of the name? If yes, the metaphor is working.

**Good metaphors:**
- Sentry (a guard → error monitoring)
- Docker (dock worker → container management)
- Vault (secure room → secret storage)
- Terraform (shaping land → shaping infrastructure)

**Bad "metaphors" (actually descriptions):**
- SiteWatch (category + generic action)
- CloudGuard (category + generic modifier)
- DataSync (two generic tech words)

The difference: a metaphor borrows meaning from another domain. A description just labels the category. Metaphors create a bridge between something the listener already understands and something new. Descriptions just point.

---

## 2. Real Words Dominate

Real-word brand names have approximately **68.8% recall vs 38.1% for invented names** in testing. The brain follows the path of least resistance — a known word repurposed beats a coined word almost every time.

This doesn't mean invented words are bad. It means they need to work harder. An invented word must be short, phonetically natural, and rooted in recognizable morphemes.

**Strongest tier — real words:**
Sentry, Vault, Linear, Stripe, Notion, Slack, Discord, Spark, Craft, Bear

**Strong tier — invented but grounded:**
Figma (from "figure"), Grafana (from "graph"), Vercel (from "versatile" + "accelerate"), Twilio (from "twilio" — twilight zone of telephony)

**Weak tier — invented without roots:**
Syncora, Brightrix, Nexagen — invented, no root recognition, interchangeable with each other

The spectrum matters. If you invent a word, anchor it in morphemes people already know. "Grafana" works because everyone sees "graph" in it. "Nexagen" works for nobody because "nexa" isn't a morpheme — it's noise.

---

## 3. Compound Words: Both Halves Must Earn Their Place

When combining two words, each component must add specific understanding or imagery. If one half is generic filler, cut it.

**The test:** Cover one half of the compound. Does the remaining half tell you something specific? Do the same for the other half. If either half is interchangeable with dozens of alternatives without changing the meaning, it's filler.

**Good compounds:**
- Datadog (data + dog — both specific, both add imagery)
- PagerDuty (pager + duty — both concrete, both connect to the user's experience)
- Wordfence (word + fence — both specific to the domain)
- Mailchimp (mail + chimp — both concrete, unexpected combination makes it stick)

**Bad compounds:**
- CloudSync (cloud = filler, could be replaced with any tech buzzword)
- SmartHub (smart = filler, hub = filler — neither half carries meaning)
- AutoFlow (auto = filler, flow = filler)
- DataBridge (data = obvious, bridge = generic)

**Pattern to watch:** If the first word is "Smart," "Auto," "Cloud," "AI," "Data," "Cyber," "Next," or "Digital" — you're almost certainly looking at filler. These words have been prepended to thousands of products and now communicate nothing.

---

## 4. Length Sweet Spot

**Syllables:**
- 1-2 syllables is optimal for memorability and daily use
- 3 syllables is the practical maximum for a single word
- Names over 3 syllables get shortened in use (Kubernetes → K8s, application → app)

**Characters:**
- 4-6 characters is the ideal range for a single word
- For invented words, 7 characters maximum — you're asking people to learn a new word, so keep the cognitive load low
- The shorter the name, the more tolerant people are of unfamiliarity

**The shortening test:** If people will inevitably abbreviate your name, either pick a name that doesn't need shortening or make sure the natural abbreviation is good. "Kubernetes" becomes "K8s" which works. "Infrastructure Platform" becomes "IP" which is already taken by all of networking.

---

## 5. The Phone Test

Say the name once to someone. They should be able to:

1. **Repeat it back** correctly
2. **Spell it** (or get close enough to search for it)
3. **Remember it** 24 hours later

If the name requires spelling out letter by letter, it's broken. Names that work visually but fail audibly will frustrate everyone who has to say them in meetings, podcasts, recommendations, and word-of-mouth.

**Pass:** Stripe, Notion, Sentry, Figma, Linear
**Fail:** DVRST, RXPTΨDE, Lyft (people will try "Lift" first), Flickr (people will try "Flicker" first)

The misspelling trend (dropping vowels, swapping letters) worked for a brief window around 2010 when the novelty factor offset the confusion. That window is closed. Today, a misspelled name signals "the real spelling was taken."

---

## 6. Names Tell Stories

Every name should have a **15-second origin story** — a concise explanation of why it's called that. This serves three purposes:

1. **Internal alignment:** The team understands the name's intent
2. **Marketing leverage:** Origin stories are natural PR/content material
3. **Quality signal:** If you can't explain why a name fits, it probably doesn't

**Good stories:**
- "Sentry — like a guard posted to watch for trouble. That's what the product does: it watches your code for errors."
- "Kafka — named after Franz Kafka, who wrote about complex bureaucratic systems. The platform handles complex data pipelines."
- "Docker — a dock worker loads and unloads cargo from ships. Docker loads and unloads applications from containers."

**No story (red flag):**
- "We picked NexaVault because it sounds modern and secure." (That's a vibe, not a story.)
- "We brainstormed for two hours and this one felt right." (Feelings aren't stories.)

If you can't tell a 15-second story, go back to metaphor exploration.

---

## 7. Distinctiveness Over Descriptiveness

A name's job is to be **memorable**, not to describe the product category. The product itself, the tagline, the website, and the onboarding experience handle description. The name handles recall.

Descriptive names feel safe but they're invisible. They blend into the category instead of standing above it.

**Distinctive:** Slack (for team communication), Discord (for group chat), Notion (for notes/docs)
**Descriptive:** TeamChat, GroupVoice, NotePad Pro

The distinctive names tell you nothing about the category at first glance — and that's fine. After one encounter with the product, the name becomes permanently linked to what it does. The descriptive names never achieve that link because they're not distinct enough to stick.

**Exception:** When the product IS the category (e.g., a niche developer tool), a slightly descriptive compound can work if it's concrete enough. "Wordfence" describes what it does (fence for WordPress) but the metaphor is concrete enough to stick.

**WordPress ecosystem note:** The wp.org plugin directory is search-driven — users find plugins by searching for their function, not by brand awareness. This creates pressure toward descriptive names. Resist it. The most memorable WordPress plugins (Yoast, Elementor, Jetpack, Akismet) are distinctive, not descriptive. Their discoverability comes from the tagline and description fields, not from keyword-stuffing the name.

**The "AI" prefix trap:** "AI" in a product name follows the same lifecycle as "e-", "i", and "Cloud" before it. It signals the era, not the product. In 2025, "AI" feels current. By 2028, it will feel like "e-" feels today — technically accurate but redundantly obvious. Build your name to outlast the buzzword cycle.

The test: remove "AI" from the name. Does it still work? If yes, you didn't need it. If the name collapses without "AI", the name was never strong enough.

---

## Summary Hierarchy

When evaluating any name candidate, check these in order. If it fails an earlier criterion, later ones don't matter:

1. **Does it tell a story?** (Even a one-sentence one)
2. **Does it plant a concrete image?**
3. **Does it sound right when spoken aloud?**
4. **Can people spell it after hearing it once?**
5. **Is it short enough to stick?**
6. **Is it distinct from competitors?**

If a name fails #1, nothing else matters. Go back to the metaphor.
